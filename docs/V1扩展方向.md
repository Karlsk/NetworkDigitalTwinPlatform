# V1 扩展方向

> 本文档记录 MVP 之后 V1 阶段的预留扩展方向，确保 MVP 架构不阻碍未来演进。

## 1. 本体继承机制 (extends)

**问题**：随着网络类型增多（SRv6、EVPN、ISIS、BGP、MPLS、VPN、Alarm、Ticket、Change），EntityType 会爆炸，每个类型独立存在，查询和分析需要枚举所有类型。

**V1 方案**：Schema EntityType 增加 `extends` 字段，支持类型继承。

### 类型层级设计

```
Resource (基类)
├── Device
├── Interface
├── Port
└── Link

Service (基类)
├── EVPN_Instance
├── Network_Slice
├── SRv6Policy
└── MPLS_VPN

Event (基类)
├── Alarm
├── Ticket
└── Change
```

### Schema 示例

```yaml
# ontology/device.yaml
metadata:
  name: Device
spec:
  extends: Resource        # ⭐ 继承 Resource 基类
  identity:
    stableKeys: [serial_number]
  uriTemplate: "device:{serial_number}"
  ...

# ontology/srv6_policy.yaml
metadata:
  name: SRv6Policy
spec:
  extends: Service         # ⭐ 继承 Service 基类
  identity:
    stableKeys: [policy_id]
  uriTemplate: "srv6:{policy_id}"
  ...
```

### Neo4j 多 Label 实现

```cypher
-- 创建节点时带多个 Label (基类 + 具体类型)
CREATE (n:Resource:Device {_db: $_db, uri: $uri, ...})

-- 查询基类自动覆盖所有子类型
MATCH (n:Resource) WHERE n._db = $_db RETURN n
MATCH (n:Service) WHERE n._db = $_db RETURN n
MATCH (n:Event) WHERE n._db = $_db RETURN n
```

### 对各层的影响

| 层 | 变化 |
|----|------|
| **GraphDB** | `CREATE` 语句使用多 Label (`:Resource:Device`) |
| **MCP 查询** | `query_topology` 可按基类查询，自动包含子类型 |
| **Impact Engine** | 爆炸半径按基类遍历，自动覆盖子类型 |
| **RCA** | 按 Event 基类查，Alarm/Ticket/Change 统一处理 |

### MVP 不阻碍的原因

- MVP 的 Label 直接使用 EntityType name（如 `:Device`），V1 只需改为多 Label
- GraphModel 的 `Node.Label` 字段已存在，V1 改为 `Labels []string` 即可
- Schema YAML 结构兼容，只需新增 `extends` 字段

## 2. 语义映射规则外置

**问题**：MVP 阶段映射规则（uriTemplate、fieldMapping、normalize）内嵌在 Schema EntityType 中。当规则膨胀到 Schema 无法承载时，需要拆分。

**V1 方案**：拆分为独立的 `configs/mapping_rules.yaml`。

```
MVP:  Schema YAML (包含 uriTemplate + normalize)  → 一个文件搞定
V1:   Schema YAML + Mapping Rules YAML            → 规则太多时拆出来
```

### 触发条件

- 单个 EntityType 的 normalize 规则超过 10 条
- 需要跨 EntityType 的共享规则（如接口名标准化规则）
- 规则需要热加载（不重启服务）

---

## 3. 多源数据集成

**问题**：MVP 只有 Mock Connector，V1 需要接入真实数据源。

**V1 方案**：

| 数据源 | Connector | 职责 (EntityType) |
|--------|-----------|-------------------|
| Netbox | `netbox` | Device, Interface, Link (静态资产) |
| 控制器 | `controller` | Device_Status, Telemetry (动态状态) |
| CMDB | `cmdb` | Device_Asset, Change (资产管理) |

### 数据源职责隔离原则

- 每个数据源管不同的 EntityType 或字段域
- **同一字段只由一个数据源负责**，从根上避免冲突
- 如果需要同一字段多源报告，用 `priority` 配置优先级

```yaml
# configs/connectors.yaml (V1)
connectors:
  - name: netbox
    type: rest
    priority: 2
    entity_types: [Device, Interface, Link]
  - name: controller
    type: rest
    priority: 1        # 动态状态优先级更高
    entity_types: [Device_Status]
```

## 4. Kafka 事件流

**问题**：MVP 使用内存 Channel 缓冲 Webhook 事件，进程重启会丢失消息。

**V1 方案**：Channel 替换为 Kafka consumer。

```
MVP:  Webhook → Channel → 消费者协程
V1:   Kafka → Consumer Group → 消费者协程
```

### 升级路径

- Channel 适配层保留，内部实现从 channel 改为 Kafka consumer
- SyncEvent 结构不变
- 获得：持久化 + 重试 + 多消费者并行

---

## 5. PostgreSQL 元数据存储

**问题**：MVP 阶段元数据（快照记录、同步历史、连接器配置）都存在内存或 YAML 文件中。

**V1 方案**：引入 PostgreSQL 存储结构化元数据。

| 表 | 用途 |
|----|------|
| `snapshots` | 快照元数据 (名称、时间、节点数、YAML 路径) |
| `sync_logs` | 同步历史 (时间、类型、结果、warnings) |
| `connectors` | 连接器配置 (名称、类型、优先级、状态) |
| `schema_versions` | Schema 版本历史 |

## 6. 分析引擎

**问题**：MVP 不包含分析引擎，V1 需要实现三大核心能力。

### Impact Engine (影响分析)

```go
// 输入: 故障节点 URI
// 输出: 爆炸半径 (受影响的节点和关系列表)
func (e *ImpactEngine) Analyze(ctx, nodeURI string) (*ImpactResult, error)
```

- 基于图遍历算法 (BFS/DFS)
- 利用本体继承，按基类查询自动覆盖子类型

### RCA Engine (根因分析)

```go
// 输入: 告警列表
// 输出: 根因节点 + 传播路径
func (e *RCAEngine) Analyze(ctx, alarms []Alarm) (*RCAResult, error)
```

- 确定性算法 (图上的因果推理)
- 利用 Event 基类统一处理 Alarm/Ticket/Change

### Simulation Engine (仿真)

```go
// 输入: 操作计划 (如"下线 Device X")
// 输出: 预期影响 (哪些 Service 受影响)
func (e *SimulationEngine) Simulate(ctx, plan *OperationPlan) (*SimResult, error)
```

- 基于 Neo4j 快照沙盒 (CloneDB → 在逻辑 DB 上模拟操作)
- 不影响 default DB

## 9. 迁移到 Neo4j 企业版

**V1+ 方案**：如果未来迁移到 Neo4j Enterprise Edition（支持真正多 DB）：

- 驱动层封装了 `_db` 细节，业务代码无感知
- 只需改驱动层实现：`_db` 属性过滤 → 真正的多 DB 连接
- Cypher 模板中的 `$_db` 变量可以移除
- 复合索引 `(_db, uri)` 可以简化为 `(uri)`
