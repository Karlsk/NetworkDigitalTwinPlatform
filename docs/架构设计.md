# 基于本体论的网络数字孪生系统 — MVP 架构方案

## Context

解决 SRv6 网络运维三大痛点：数据孤岛（控制器 vs 数据中台割裂）、缺乏语义认知（图数据库无语义）、高阶运维断层（RCA 依赖专家经验）。通过 YAML Schema Registry 定义网络本体，插件式 Connector 采集多源数据，归一化后构建 Neo4j 图孪生体，通过 MCP 暴露给外部 Agent 平台。

## 核心决策汇总

| 决策项 | 选型 | 理由 |
|--------|------|------|
| 本体定义 | YAML Schema Registry (K8s CRD 风格) | 单一数据源，新增实体零代码，开发者友好 |
| 快照格式 | YAML Snapshot + Neo4j 逻辑多DB | YAML 归档导出，Neo4j 内可查询/对比/沙盒 |
| 图数据库 | Neo4j CE + 驱动层逻辑多DB | CE 不支持多DB，通过 `_db` 属性实现逻辑隔离 |
| 元数据存储 | MVP 本地文件，V1 引入 PostgreSQL | MVP 保持简洁，后续按需引入 |
| 智能层 | 纯算法引擎 + MCP | RCA/Simulation/Impact 确定性算法，不依赖 LLM |
| n10s | 不依赖 | 直接用 Cypher 构建图，简化部署 |
| Connector | 插件式 Connector + Registry | MVP 只实现 Mock，架构预留扩展点 |
| 项目名 | `network-digital-twin` | Go module: `gitlab.com/pml/network-digital-twin` |

## 数据流管线 (Pipeline)

系统核心数据流采用**分层 IR（中间表示）** 设计，每层只关心自己的转换，通过显式契约解耦：

```
Connector
  → Resource                          (原始数据：Kind + ID + Properties)
  → Normalizer                         (读 Schema EntityType → 字段标准化 + URI 生成)
  → NormalizedResource                 (归一化节点：Kind + URI + Properties，不含关系)
  → GraphAssembler  ⭐                 (读 relationFields 映射 + RelationType → 推导关系)
  → GraphModel (Nodes + Relations)     (纯图 IR：Label + Props + Edges，与数据源/Schema 无关)
  → GraphDB                            (封装 Cypher 生成 + 执行)
  → Neo4j
```

### 各层职责边界

| 层 | 输入 | 输出 | 依赖 Schema? | 职责 |
|---|------|------|-------------|------|
| Connector | 外部数据源 | Resource | ❌ | 采集原始数据 |
| Normalizer | Resource | NormalizedResource | ✅ EntityType | 字段标准化、URI 生成（只处理节点，不处理关系） |
| GraphAssembler | NormalizedResource | GraphModel | ✅ EntityType.relationFields + RelationType | 按 relationFields 映射推导关系，组装图节点+图边 |
| GraphDB | GraphModel | 数据库操作 | ❌ | Cypher 生成 + 执行，封装图数据库细节 |

### Schema Registry 消费者

| 消费者 | 读取 Schema 的哪部分 | 用途 |
|--------|---------------------|------|
| Normalizer | EntityType | 属性类型、枚举、默认值 → 校验+标准化字段 |
| GraphAssembler | EntityType.relationFields + RelationType | relationFields 告知哪些字段推导关系，RelationType 校验源/目标类型 |
| GraphDB | **不读 Schema** | 只接收 GraphModel，生成 Cypher + 执行 |
| Validator | EntityType | 校验最终数据合法性 |

### GraphModel (IR) 定义

```go
// GraphModel 是纯图元素的中间表示，与数据源和 Schema 完全解耦
type GraphModel struct {
    Nodes     []Node
    Relations []Relation
}

type Node struct {
    Label  string         // 图节点标签，如 "Device"
    URI    string         // 唯一标识
    Props  map[string]any // 属性键值对
}

type Relation struct {
    Type   string         // 关系类型，如 "HAS_INTERFACE"
    From   string         // 源节点 URI
    To     string         // 目标节点 URI
    Props  map[string]any // 关系属性（可选）
}
```

**关键设计**：GraphDB 只接收 `GraphModel`，不知道 Schema 的存在。这使得：
- 修改 Schema（如新增 EntityType）不影响 GraphDB
- 可通过 `BuildCypher` 方法预览生成的 Cypher（测试/audit）
- 未来替换图数据库只需新建 GraphDB 实现

## 核心模块设计

### Task 1: Schema Registry (本体定义)

**文件**: `ontology/*.yaml`, `internal/schema/`

YAML CRD 风格的实体类型定义，系统启动时自动加载到 SchemaRegistry。

```yaml
# ontology/device.yaml
apiVersion: twin.io/v1
kind: EntityType
metadata:
  name: Device
  labels: [Resource, Network]
spec:
  identity:                              # ⭐ 不可变身份标识 (URI 永久不变)
    stableKeys: [serial_number]          # 序列号/机箱MAC等不会变的字段
  uriTemplate: "device:{serial_number}"  # ⭐ URI 生成模板 (必须基于 stableKeys)
  fieldMapping:                          # ⭐ 字段映射 (源字段 → 标准字段)
    mgmt_ip: management_ip
    hw_model: model
  normalize:                             # ⭐ 字段标准化规则
    - field: hostname
      pattern: " "
      replace: "_"
  relationFields:                        # ⭐ 关系字段映射 (Properties 字段 → RelationType)
    interfaces:
      relationType: HAS_INTERFACE
    upstream_links:
      relationType: CONNECTS_TO
  properties:
    serial_number:
      type: string
      required: true
    hostname:
      type: string
      required: true
    vendor:
      type: string
    model:
      type: string
    management_ip:
      type: string
    chassis_mac:
      type: string
    status:
      type: string
      enum: [Up, Down, Maintenance]
      default: "Up"
    device_type:
      type: string
      enum: [Core, Edge, Access]
```

**设计原则**：
- `identity.stableKeys` 必须是**不可变**的标识（如 `serial_number`、`chassis_mac`、`asset_id`）
- `uriTemplate` **只能基于 stableKeys** 生成，不能使用可变属性
- `hostname`、`management_ip` 等可变属性只是**节点属性**，不是身份标识
- 这样即使设备改名、换 IP，URI 仍然不变，所有关系和快照历史保持有效
- `relationFields` 声明 Properties 中哪些字段推导关系，**新增关系类型只需修改 YAML，零代码**

```yaml
# ontology/relations.yaml
apiVersion: twin.io/v1
kind: RelationType
metadata:
  name: HAS_INTERFACE
spec:
  source: [Device]
  target: [Interface]
---
kind: RelationType
metadata:
  name: RUNS_ON_INTERFACE
spec:
  source: [SRv6_Policy]
  target: [Interface]
---
kind: RelationType
metadata:
  name: CARRIED_BY
spec:
  source: [EVPN_Instance]
  target: [SRv6_Policy]
---
kind: RelationType
metadata:
  name: BELONGS_TO_SLICE
spec:
  source: [EVPN_Instance]
  target: [Network_Slice]
---
kind: RelationType
metadata:
  name: OCCURRED_ON
spec:
  source: [Alarm]
  target: [Interface]
```

### Task 3: 归一化引擎

**文件**: `internal/normalizer/`

职责：读取 Schema **EntityType** 中的 `uriTemplate`、`fieldMapping`、`normalize` 规则，将 Connector 输出的原始 Resource 进行字段标准化，产出 `NormalizedResource`（只包含节点信息，不包含关系）。

**Normalizer 从 Schema 读取的规则**：

| Schema 字段 | Normalizer 用途 |
|------------|----------------|
| `uriTemplate` | 生成标准本体 URI（如 `device:{serial_number}`） |
| `fieldMapping` | 字段名映射（如 `mgmt_ip` → `management_ip`） |
| `normalize` | 字段值标准化（如空格替换为下划线） |
| `properties` | 属性类型校验、枚举校验、默认值填充 |
| `relationFields` | **不读取**，关系字段保留在 Properties 中传递给 GraphAssembler |

**设计原则**：Normalizer **只处理节点**（字段标准化 + URI 生成），**不处理关系**。关系由 GraphAssembler 从 Properties 推导。

```go
type Normalizer struct {
    registry  schema.SchemaRegistry
}

// Normalize: Resource → NormalizedResource (只包含节点信息)
// 1. 从 Schema 读取 EntityType 的 uriTemplate, fieldMapping, normalize 规则
// 2. fieldMapping: 映射字段名
// 3. normalize: 标准化字段值
// 4. uriTemplate: 生成 URI
// 5. properties: 类型校验 + 默认值填充
func (n *Normalizer) Normalize(resource Resource) (*NormalizedResource, error)

// NormalizedResource: 归一化后的节点记录 (不含关系)
// 关系字段 (如 interfaces: ["iface:..."]) 保留在 Properties 中，
// 由 GraphAssembler 读取 relationFields 映射后推导为图边
type NormalizedResource struct {
    Kind       string                 // 实体类型，如 "Device"
    URI        string                 // 标准本体 URI (由 uriTemplate 生成)
    Properties map[string]any         // 标准化后的属性（类型已校验，含原始关系字段）
}
```

**设计原则**：所有映射规则内嵌在 Schema EntityType 中（单一数据源），MVP 阶段不引入独立的 Mapping Rules 配置文件。当规则膨胀到 Schema 无法承载时（V1 阶段），再拆分为独立的 `configs/mapping_rules.yaml`。

### Task 4: GraphAssembler (IR 层) ⭐

**文件**: `internal/assembler/`

职责：读取 Schema **EntityType.relationFields** + **RelationType**，将一批 `NormalizedResource` 组装为纯图元素 `GraphModel`。此层是数据流的**关键解耦点**：上游是归一化节点记录，下游是纯图模型。

**核心职责**：
1. **节点转换**：每个 `NormalizedResource` → 一个 `Node`（过滤掉关系字段）
2. **关系推导**：读 `relationFields` 映射 → 从 Properties 提取关系字段 → 查 RelationType 校验 → 生成 `Relation`

```go
type GraphAssembler struct {
    registry schema.SchemaRegistry
}

// Assemble: []NormalizedResource → GraphModel (纯图节点+图边，与 Schema/数据源无关)
func (a *GraphAssembler) Assemble(resources []NormalizedResource) (*GraphModel, error)

// GraphModel 是纯图元素的中间表示
type GraphModel struct {
    Nodes     []Node
    Relations []Relation
}

type Node struct {
    Label  string         // 图节点标签，如 "Device"
    URI    string         // 唯一标识
    Props  map[string]any // 属性键值对（已类型化，不含原始关系字段）
}

type Relation struct {
    Type   string         // 关系类型，如 "HAS_INTERFACE"
    From   string         // 源节点 URI
    To     string         // 目标节点 URI
    Props  map[string]any // 关系属性（可选）
}
```

**组装规则**：

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | NormalizedResource → Node | Label=Kind, URI=URI, Props=Properties（过滤掉关系字段） |
| 2 | 读 Schema relationFields | 如 `interfaces → HAS_INTERFACE` |
| 3 | 从 Properties 提取关系字段 | 如 Device.Properties.interfaces: ["iface:SN12345_GE1/0/1"] |
| 4 | 查 Schema RelationType 校验 | 如 HAS_INTERFACE: source=Device, target=Interface |
| 5 | 生成 Relation | From=Device.URI, To="iface:SN12345_GE1/0/1", Type="HAS_INTERFACE" |

**关系推导示例**：

```
输入: NormalizedResource {
    Kind: "Device",
    URI: "device:SN12345",
    Properties: {
        serial_number: "SN12345",
        hostname: "Router_Core_01",
        interfaces: ["iface:SN12345_GE1/0/1", "iface:SN12345_GE1/0/2"]  // 关系字段
    }
}

Schema EntityType.relationFields:
    interfaces → HAS_INTERFACE       ← 映射声明

Schema RelationType:
    HAS_INTERFACE: source=Device, target=Interface  ← 类型校验

输出: GraphModel {
    Nodes: [Node{Label: "Device", URI: "device:SN12345", Props: {hostname: "..."}}],
    Relations: [
        Relation{Type: "HAS_INTERFACE", From: "device:SN12345", To: "iface:SN12345_GE1/0/1"},
        Relation{Type: "HAS_INTERFACE", From: "device:SN12345", To: "iface:SN12345_GE1/0/2"}
    ]
}
```

**关键设计**：`relationFields` 让 GraphAssembler 知道哪个 Properties 字段对应哪个 RelationType，**新增关系类型只需修改 YAML，零代码**：

```yaml
# 新增 SRv6Policy 关系 — 只改 YAML
spec:
  relationFields:
    interfaces:
      relationType: HAS_INTERFACE
    upstream_links:
      relationType: CONNECTS_TO
    srv6_policies:                    # ⭐ 新增
      relationType: RUNS_ON_INTERFACE
```

**设计原则**：
- Normalizer **只处理节点**（字段标准化 + URI 生成），读 EntityType 属性定义
- GraphAssembler **只处理图结构**（节点转换 + 关系推导），读 EntityType.relationFields + RelationType
- 职责清晰，新增实体/关系类型只需修改 YAML，零代码

**GraphModel 校验（孤儿边检测）**：

Assembler 产出 GraphModel 后，校验关系目标节点是否存在。孤儿边**跳过 + Warn**，不阻断同步。

```
场景:
  Device R1 引用 interfaces: ["iface:SN12345_GE1/0/2"]
  但该 Interface 不在当前批次中
  → 关系指向不存在的节点

处理:
  校验发现孤儿边 → log.Warn("orphan edge skipped: HAS_INTERFACE → iface:SN12345_GE1/0/2")
  → 跳过该关系，继续处理其他节点和关系
  → SyncResult.OrphanEdgesSkipped++ (可观测)

不阻断原因:
  一个 Interface 缺失不应该导致整个同步批次失败
```

```go
// GraphAssembler.Assemble() 内部校验
func (a *GraphAssembler) Assemble(resources []NormalizedResource) (*GraphModel, []ValidationWarning, error) {
    // 1. 节点转换
    // 2. 关系推导
    // 3. 校验: 检查关系目标节点是否存在于 Nodes 中
    //    → 不存在: 跳过 + 返回 Warning (不报错)
    ...
}

type ValidationWarning struct {
    Type    string  // "orphan_edge"
    Detail  string  // "HAS_INTERFACE: device:SN12345 → iface:SN12345_GE1/0/2"
}

// SyncResult 包含校验统计
type SyncResult struct {
    NodesCreated       int
    RelationsCreated  int
    OrphanEdgesSkipped int  // ⭐ 孤儿边计数 (可观测)
    Warnings           []ValidationWarning
}
```

### Task 5: 图数据库驱动 (Neo4j 逻辑多DB)

**文件**: `internal/graph/`

职责：GraphDB 接口封装图数据库的所有操作，Cypher 生成逻辑内聚在实现中。提供 `BuildCypher` 方法用于预览、测试和 audit。

```go
// GraphDB interface: 所有操作带 db 参数
type GraphDB interface {
    Ping(ctx context.Context) error
    Close() error

    // 全量同步: 批量 CREATE (先 ClearDB 再 BulkCreate)
    BulkCreate(ctx context.Context, db string, nodes []Node, rels []Relation) error

    // 增量同步: UPSERT
    // 节点: MERGE + SET d += $props (属性增量合并)
    // 关系: MERGE (增量更新，不删旧关系)
    Upsert(ctx context.Context, db string, nodes []Node, rels []Relation) error

    // 增量同步: 删除指定关系 (仅收到明确删除事件时调用)
    DeleteRelations(ctx context.Context, db string, rels []Relation) error

    // 增量同步: 按 URI 删除节点和关系 (节点下线时调用)
    DeleteByURIs(ctx context.Context, db string, uris []string) error

    // 查询
    Query(ctx context.Context, db string, cypher string, params map[string]any) ([]map[string]any, error)

    // Cypher 预览: 生成 Cypher 语句但不执行 (用于测试/audit/调试)
    BuildCypher(action string, db string, nodes []Node, rels []Relation, uris []string) (string, map[string]any)
    // action: "create", "upsert", "delete", "delete_relations"

    // 逻辑 DB 管理
    ClearDB(ctx context.Context, db string) error
    CloneDB(ctx context.Context, from, to string) error
    ListDBs(ctx context.Context) ([]string, error)
    HasDB(ctx context.Context, db string) (bool, error)
}
```

**设计原则**：
- Cypher 生成内聚在 GraphDB 实现中，无独立 Builder 层
- `BuildCypher` 方法用于预览生成的 Cypher（测试/audit/调试），与 `Query` 组合使用
- 换图数据库 → 新建 GraphDB 实现，内部生成对应查询语言

#### 驱动层 `_db` 强制注入机制

Neo4j CE 不支持多 DB，通过 `_db` 属性模拟逻辑多 DB。为消除"忘记加 WHERE _db"导致查询污染的风险，在驱动层**强制注入** `_db` 过滤条件。

**设计原则**：业务代码不关心 `_db` 过滤，驱动层自动保证。

```go
type neo4jClient struct {
    driver neo4j.DriverWithContext
}

func (c *neo4jClient) Query(ctx context.Context, db string, cypher string, params map[string]any) ([]map[string]any, error) {
    // 驱动层强制注入 _db 参数
    params["_db"] = db
    
    // Cypher 模板统一使用 $_db 变量
    // 业务层写: MATCH (d:Device) WHERE d._db = $_db RETURN d
    // 驱动层自动保证 $_db 被注入
    ...
}

func (c *neo4jClient) Update(ctx context.Context, db string, label, uri string, props map[string]any) error {
    // 驱动层自动注入 _db
    props["_db"] = db
    ...
}
```

**Cypher 模板规范**：

```cypher
-- 所有查询模板必须使用 $_db 变量（驱动层注入）
-- 正确 ✅
MATCH (d:Device) WHERE d._db = $_db RETURN d

MATCH (d:Device)-[:HAS_INTERFACE]->(i:Interface)
WHERE d._db = $_db AND i._db = $_db
RETURN d, i

-- 错误 ❌ (驱动层会报错或静默注入，取决于实现)
MATCH (d:Device) RETURN d  -- 缺少 _db 过滤
```

**索引策略**：

```cypher
-- Composite index 确保 _db 过滤高效
CREATE INDEX device_db_uri FOR (d:Device) ON (d._db, d.uri);
CREATE INDEX interface_db_uri FOR (i:Interface) ON (i._db, i.uri);
-- 每个实体类型都需要创建 (_db, uri) 复合索引
```

**安全保障**：

| 风险 | 驱动层应对 |
|------|-----------|
| 忘记加 WHERE _db | 驱动层自动注入 `$_db` 参数 |
| 跨 DB 查询污染 | 驱动层强制所有操作绑定 db 参数 |
| 索引失效 | 强制使用 `(_db, uri)` 复合索引 |
| 迁移到 Neo4j 企业版 | 驱动层封装 `_db` 细节，业务代码无感知，迁移时只需改驱动层实现 |

### Task 6: 快照管理

**文件**: `internal/snapshot/`

**设计原则**：Neo4j 是唯一 source of truth，YAML 是永久归档产物。

```
Create 快照:    default DB → 导出 YAML 归档文件
                (Neo4j 不写任何东西，YAML 永久保留不删除)

查询/对比快照:  先查 Neo4j 是否已有该逻辑 DB
                → 有: 直接使用
                → 没有: 从 YAML 恢复到 Neo4j 逻辑 DB (名为 snapshot name)

清理策略:      只保留"最近使用时间"最新的 5 个 Neo4j 逻辑 DB
                超过的自动 ClearDB (YAML 归档文件不删除)
```

**GraphLock 并发保护**：

Restore 和 FullSync 操作 default DB 时，必须持有写锁，防止增量同步（Webhook）同时写入导致脏图。

```
问题场景:
  Restore 执行: ClearDB(default) → CloneDB(snap, default)
  Webhook 同时到达: Upsert(default, new_data)
  结果: default 混合快照数据 + 最新数据 = 脏图 ❌

解决方案: GraphLock (sync.RWMutex)
  Restore/FullSync 持有写锁期间:
    → 增量同步 (Upsert/DeleteByURIs) 被阻塞或拒绝
    → 只读查询不受影响 (读锁兼容)
```

```go
type GraphLock struct {
    mu sync.RWMutex
}

// Lock: Restore/FullSync 使用 (排他锁，阻塞增量写入)
func (l *GraphLock) Lock()
func (l *GraphLock) Unlock()

// RLock: 只读查询使用 (共享锁，不阻塞其他读)
func (l *GraphLock) RLock()
func (l *GraphLock) RUnlock()
```

**GraphLock 使用场景**：

| 操作 | 锁类型 | 说明 |
|------|--------|------|
| Restore | 写锁 (Lock) | ClearDB + CloneDB 期间禁止增量写入 |
| FullSync | 写锁 (Lock) | ClearDB + BulkCreate 期间禁止增量写入 |
| IncrementalSync | 写锁 (Lock) | Upsert/DeleteByURIs 期间禁止其他写入 |
| MCP Query | 读锁 (RLock) | 查询期间允许其他读，阻塞写 |
| Snapshot.Create | 读锁 (RLock) | 导出期间允许其他读 |

```go
type SnapshotManager struct {
    graph      graph.GraphDB
    lock       *GraphLock  // ⭐ 并发保护
    snapDir    string      // YAML 快照文件存储目录 (永久归档)
    maxActive  int         // Neo4j 中最多保留的逻辑 DB 数量，默认 5
}

// Create: 导出 default DB 为 YAML 归档文件 (不写 Neo4j)
func (m *SnapshotManager) Create(ctx context.Context, name string) (*SnapshotMeta, error)
// 1. lock.RLock()
// 2. 查询 default DB 全量数据
// 3. 序列化为 YAML 文件 → 写入 snapDir/{name}.yaml
// 4. lock.RUnlock()
// 5. 记录元数据 (时间戳、节点数、关系数)

// EnsureLoaded: 确保快照已加载到 Neo4j 逻辑 DB (懒加载)
func (m *SnapshotManager) EnsureLoaded(ctx context.Context, name string) error
// 1. 检查 Neo4j 是否已有名为 name 的逻辑 DB
// 2. 有 → 直接使用
// 3. 没有 → 读取 YAML 文件 → 写入 Neo4j 逻辑 DB
// 4. 触发清理: 如果活跃逻辑 DB 数 > maxActive，清理最久未使用的

// Restore: 将指定快照恢复为 default DB (持有写锁)
func (m *SnapshotManager) Restore(ctx context.Context, name string) error
// 1. lock.Lock()                ← 排他锁，阻塞增量同步
// 2. EnsureLoaded(name)         → 确保快照已在 Neo4j 逻辑 DB 中
// 3. ClearDB("default")
// 4. CloneDB(name, "default")
// 5. lock.Unlock()              ← 释放锁

func (m *SnapshotManager) List(ctx context.Context) ([]SnapshotMeta, error)
// 列出所有 YAML 归档 (永久保存)

func (m *SnapshotManager) Delete(ctx context.Context, name string) error
// 1. ClearDB(name)             → 清理 Neo4j 逻辑 DB (如果存在)
// 2. YAML 归档文件保留不删除

func (m *SnapshotManager) Diff(ctx context.Context, a, b string) (*SnapshotDiff, error)
// 1. EnsureLoaded(a) + EnsureLoaded(b)
// 2. Cypher 差集查询两个 Neo4j 逻辑 DB

// cleanup: 清理 Neo4j 中超过 maxActive 的逻辑 DB (按最近使用时间排序)
func (m *SnapshotManager) cleanup(ctx context.Context) error
// 1. 列出所有逻辑 DB + 最近使用时间
// 2. 保留最新的 maxActive 个
// 3. 其余 ClearDB (YAML 归档不动)
```

### Task 9: 业务编排层

**文件**: `internal/service/`

```go
// SyncService: 编排完整的数据同步流水线
type SyncService struct {
    registry   *connector.ConnectorRegistry
    normalizer *normalizer.Normalizer
    assembler  *assembler.GraphAssembler  // ⭐ IR 层
    graph      graph.GraphDB
    lock       *snapshot.GraphLock        // ⭐ 与 SnapshotManager 共享同一把锁
}

// FullSync: 全量同步 (定时触发，持有写锁)
// 1. lock.Lock()                    ← 排他锁，阻塞增量同步和恢复
// 2. ClearDB("default")
// 3. 所有 Connector.Collect() 获取全量数据
// 4. Normalizer → Assembler → GraphDB.BulkCreate
// 5. lock.Unlock()                  ← 释放锁
func (s *SyncService) FullSync(ctx) (*SyncResult, error)

// IncrementalSync: 增量同步 (事件触发: Webhook/Kafka，持有写锁)
// 根据 event.Action 分发:
//   "update"           → Normalizer → Assembler → GraphDB.Upsert() (MERGE 节点+关系)
//   "delete"           → GraphDB.DeleteByURIs() (DETACH DELETE 节点)
//   "delete_relation"  → GraphDB.DeleteRelations() (仅删除指定关系)
// 不需要对比，Connector 直接告知变更内容
func (s *SyncService) IncrementalSync(ctx context.Context, event SyncEvent) (*SyncResult, error)
```

**并发保护说明**：
- SyncService 和 SnapshotManager **共享同一个 GraphLock 实例**
- FullSync/IncrementalSync/Restore 三个写操作互斥，同一时刻只有一个能执行
- 如果锁已被持有，新的写入请求会**阻塞等待**（或返回 503 Service Unavailable）

#### 增量同步策略

**触发方式**：事件驱动
- MVP: Webhook (外部系统推送变更事件) + Channel 缓冲
- V1: Kafka (高吞吐事件流，天然持久化)

**关键设计：Channel 缓冲防丢消息**

```
问题场景:
  Restore 持有 GraphLock 期间 (可能耗时数秒)
  Webhook 请求到达 → 如果阻塞等锁 → 外部系统超时 → 重试 → 消息重复/丢失

解决方案: Webhook → Channel → 消费者协程

  WebhookHandler
      │
      │ 立即写入 channel → 返回 202 Accepted (毫秒级)
      ▼
  SyncEvent (入队)
      │
      ▼
  消费者协程 (单协程，串行消费)
      │
      │ 等待 GraphLock 可用
      ▼
  IncrementalSync()
      │ lock.Lock()
      │ Upsert / DeleteByURIs
      │ lock.Unlock()
```

**MVP 实现**：
```go
type SyncEvent struct {
    Action      string   // "update", "delete", "delete_relation"
    EntityType  string
    Connector   string
    Data        []map[string]any  // update 时的数据
    URIs        []string          // delete 时的 URI 列表
    Relations   []Relation        // delete_relation 时的关系列表
}

type SyncService struct {
    registry   *connector.ConnectorRegistry
    normalizer *normalizer.Normalizer
    assembler  *assembler.GraphAssembler
    graph      graph.GraphDB
    lock       *snapshot.GraphLock
    eventChan  chan SyncEvent  // ⭐ 缓冲 channel，防丢消息
}

// StartConsumer: 启动消费者协程 (服务启动时调用)
func (s *SyncService) StartConsumer(ctx context.Context) {
    go func() {
        for event := range s.eventChan {
            // 等待 GraphLock → 执行 IncrementalSync → 释放锁
            s.lock.Lock()
            s.processEvent(ctx, event)
            s.lock.Unlock()
        }
    }()
}

// processEvent: 处理单个事件
func (s *SyncService) processEvent(ctx context.Context, event SyncEvent) {
    switch event.Action {
    case "update":
        // Normalizer → Assembler → GraphModel
        // GraphDB.Upsert() → MERGE 节点 + MERGE 关系
    case "delete":
        // GraphDB.DeleteByURIs() → DELETE 节点 + DETACH DELETE 关系
    case "delete_relation":
        // GraphDB.DeleteRelations() → 仅删除指定关系
    }
}

// HandleWebhook: Webhook Handler (立即返回 202)
func (s *SyncService) HandleWebhook(event SyncEvent) error {
    select {
    case s.eventChan <- event:
        return nil  // 入队成功，返回 202 Accepted
    default:
        return errors.New("event queue full")  // channel 满，返回 503
    }
}
```

**同步流程**：

```
外部系统 (Netbox/控制器)
    │
    │ Webhook POST /api/v1/sync/webhook
    │ payload: { action: "update", entity_type: "Device", data: [...] }
    ▼
WebhookHandler
    │
    │ 立即写入 eventChan → 返回 202 Accepted
    ▼
SyncEvent 入队
    │
    ▼
消费者协程 (串行)
    │
    │ 等待 GraphLock 可用
    ▼
SyncService.IncrementalSync()
    │
    ├── action = "update" / "create"
    │   → Normalizer → Assembler → GraphModel
    │   → GraphDB.Upsert("default", nodes, rels)   // MERGE + SET
    │
    └── action = "delete"
        → GraphDB.DeleteByURIs("default", uris)    // DETACH DELETE
```

**关键设计**：
- 增量同步**直接操作 default DB**，不做数据对比
- Connector 通过 Webhook payload 告知哪些实体新增/更新/删除
- Neo4j 驱动层提供 `Upsert` (MERGE) 和 `DeleteByURIs` (DETACH DELETE) 接口
- **Channel 缓冲**：Webhook 立即返回，消费者协程异步处理，GraphLock 期间不丢消息
- **单协程消费**：保证事件顺序，避免并发写入冲突
- **V1 升级路径**：Channel 替换为 Kafka consumer，持久化 + 重试机制

**Webhook Payload 格式**：

```json
{
  "action": "update",
  "entity_type": "Device",
  "connector": "netbox",
  "data": [
    { "serial_number": "SN12345", "hostname": "Router_Core_01", "status": "Down", ... }
  ]
}
```

```json
{
  "action": "delete",
  "entity_type": "Device",
  "connector": "netbox",
  "uris": ["device:SN99999"]
}
```

```json
{
  "action": "delete_relation",
  "entity_type": "Device",
  "connector": "netbox",
  "relations": [
    { "type": "HAS_INTERFACE", "from": "device:SN12345", "to": "iface:SN12345_GE1/0/2" }
  ]
}
```

**Action 语义**：

| Action | 节点处理 | 关系处理 |
|--------|---------|---------|
| `update` | MERGE + SET += | MERGE（新增，不删旧关系） |
| `delete` | DELETE 节点 + DETACH DELETE | 节点删除时自动清理所有关联关系 |
| `delete_relation` | 不涉及 | 仅删除指定关系 |

#### ID 稳定性与合并策略

**1. URI 永久不变性 (URI Immutability)**

URI 由 Schema EntityType 的 `identity.stableKeys` + `uriTemplate` 生成。`stableKeys` **必须选择不可变**的标识（如序列号、机箱 MAC、资产编号），确保：

```yaml
# ✅ 正确: 序列号不可变，URI 永久有效
spec:
  identity:
    stableKeys: [serial_number]
  uriTemplate: "device:{serial_number}"

# ✅ 正确: 机箱 MAC 不可变
spec:
  identity:
    stableKeys: [chassis_mac]
  uriTemplate: "device:{chassis_mac}"

# ❌ 错误: hostname 会变，设备改名后 URI 失效
spec:
  identity:
    stableKeys: [hostname]
  uriTemplate: "device:{hostname}"

# ❌ 错误: IP 会变，设备换 IP 后 URI 失效
spec:
  identity:
    stableKeys: [management_ip]
  uriTemplate: "device:{management_ip}"
```

**关键原则**：
- `hostname`、`management_ip`、`status` 都是**可变属性**，只是节点属性，不是身份
- `serial_number`、`chassis_mac`、`asset_id` 是**不可变身份**，用于生成永久 URI
- 设备改名、换 IP 后，URI 不变 → 所有关系、快照历史、Diff、RCA 都保持有效
- 不需要跨源 ID 映射表，只要 `stableKeys` 选对，不同 Connector 自然对齐

**2. 属性合并策略 (Property Merge)**

增量更新使用 `SET d += $props`（**增量合并**）：

```cypher
-- 只更新传入的属性，未传入的属性保持不变
MERGE (d:Device {_db: $_db, uri: $uri})
SET d += $props
```

```
第一次: { serial_number: "SN12345", hostname: "R1", status: "Up", bandwidth: 100 }
第二次: { serial_number: "SN12345", hostname: "R1", status: "Down" }  // bandwidth 未传
结果:   { serial_number: "SN12345", hostname: "R1", status: "Down", bandwidth: 100 }  ✅ 保留
```

**3. 关系合并策略 (Relation Merge)**

增量更新时，**关系增量更新**：新增关系用 MERGE，只有明确的删除事件才删除关系。

```cypher
-- 新增关系: MERGE (幂等，已存在则跳过)
MERGE (a {_db: $_db, uri: $from})-[:HAS_INTERFACE]->(b {_db: $_db, uri: $to})

-- 删除关系: 仅当收到明确删除事件时
MATCH (a {_db: $_db, uri: $from})-[r:HAS_INTERFACE]->(b {_db: $_db, uri: $to})
DELETE r
```

**设计原则**：
- Webhook `action: "update"` → 关系 MERGE（新增/保持不变，不删旧的）
- Webhook `action: "delete_relation"` → 明确删除指定关系
- 全量同步 (FullSync) 时仍然先 ClearDB 再 BulkCreate，不受影响

**4. 防重复机制 (Dedup)**

`MERGE` 基于 `(_db, uri)` 复合键，保证同一 URI 不会创建重复节点。

| 机制 | 策略 | Neo4j 实现 |
|------|------|-----------|
| URI 永久不变 | stableKeys 选不可变标识 | `uriTemplate` 生成 |
| 属性合并 | 增量合并 (SET +=) | `MERGE + SET d += $props` |
| 关系合并 | 增量更新 (MERGE) | `MERGE` 新增关系，`delete_relation` 事件明确删除 |
| 节点防重 | MERGE 基于复合键 | `MERGE (n {_db, uri})` |

## MVP 阶段目标

**数据底座优先**，完整打通核心数据流：

```
Connector(Mock) → Normalizer → GraphAssembler(IR) → GraphDB → Neo4j → 快照管理 → MCP 查询
```

### MVP 交付物

1. **Schema Registry**: 6 个 EntityType + 5 个 RelationType 的 YAML 定义 + 自动加载
2. **Connector 框架**: 接口 + Registry + Mock 实现 (3 台设备, ~20 节点)
3. **归一化引擎**: 从 Schema 读取 uriTemplate/fieldMapping/normalize 规则
4. **GraphAssembler (IR 层)**: NormalizedResource → GraphModel 组装，与 Schema 解耦的关键中间层
5. **图数据库驱动 (GraphDB)**: Cypher 生成 + 执行，含逻辑多DB、Upsert/DeleteByURIs、BuildCypher 预览
6. **同步服务**: 全量同步 (ClearDB + BulkCreate) + 增量同步 (Webhook 事件触发 + Upsert/DeleteByURIs)
7. **快照管理**: 创建/恢复/列表/删除/对比 (YAML 归档永久保留，Neo4j 逻辑 DB 懒加载)
8. **GraphLock 并发保护**: Restore/FullSync/IncrementalSync 写锁互斥，防止脏图
9. **基础 MCP 工具**: query_topology + query_snapshot + sync_data (读写分离，统一通过 Service 编排)
10. **Docker Compose**: Neo4j CE + Go 服务
11. **集成测试**: Mock 数据导入 → 查询验证 → 快照恢复验证

### MVP 不包含 (放 V1)

- RCA Engine / Impact Engine / Simulation Engine
- PostgreSQL
- 真实数据源 Connector (Netbox/Controller/CMDB)
- HTTP API (Gin)
- 可观测性 (OpenTelemetry/Prometheus)
- 定时调度 (gocron)
- 本体继承机制 (extends)
- Kafka 事件流 (替代 Channel 缓冲)

> 详见 [V1扩展方向.md](./V1扩展方向.md)

## 验证方式

1. `go build ./...` 编译通过
2. `go test ./...` 单元测试通过
3. Docker Compose 启动 Neo4j CE + Go 服务
4. Mock 数据全量同步 → Neo4j 查询验证节点/关系数
5. 快照创建 → 修改数据 → 快照恢复 → 验证数据一致
6. MCP 工具调用: get_topology 返回正确拓扑数据
7. `make test-e2e` E2E 测试套件: 完整数据流串联 + 并发保护验证
8. `make verify-all` 全量验收: 编译 + Lint + 单测 + Race 检测 + E2E

### E2E 测试覆盖范围

| 测试 | 场景 |
|------|------|
| `TestE2E_FullDataFlow` | FullSync → Query → Snapshot → IncrementalSync → Diff → Restore → 验证闭环 |
| `TestE2E_ConcurrentProtection` | Restore 期间 Webhook 事件缓冲与串行处理验证 |
| `TestE2E_SnapshotLifecycle` | Create → List → Restore → Diff 快照生命周期 |
| `TestE2E_IncrementalSync` | Upsert + DeleteByURIs + DeleteRelations 增量操作 |

### 已知限制

- `SnapshotManager.Diff` (Cypher 差集查询) 存在边界行为，推荐使用 `LocalDiff` (YAML 内存对比)
- Mock 数据产出 21 节点 + 21 关系，非早期设计文档中的 ≥30 关系
- E2E 并发测试存在时序不确定性（Neo4j 操作毫秒级完成）
