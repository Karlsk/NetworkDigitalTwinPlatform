# V1 架构设计文档

> 基于 MVP 35 项任务全部完成后制定的 V1 阶段架构方案。
>
> Go Module: `gitlab.com/pml/network-digital-twin`

## 1. 总体概况

### 1.1 V1 目标

从 MVP（Mock 数据 + 单 Label + URI 级 Diff + 薄快照服务）升级为 V1：
- **真实数据源**: Netbox/CMDB/Controller REST Connector 替代 Mock 数据
- **本体继承**: `extends` 机制支持 Neo4j 多 Label（`:Resource:Device`）
- **属性级 Diff**: 快照对比从 URI 增删扩展为逐字段属性变更检测
- **缓存审计**: SnapshotService MetaCache 优化 List 性能，AuditLog 记录操作审计

### 1.2 MVP 技术债（V1 偿还）

| 编号 | 问题 | V1 解决方案 |
|------|------|------------|
| TD-03 | SnapshotService 是薄封装 | MetaCache + AuditLog 增强 |
| TD-04 | 快照 Diff 只对比 URI 存在性 | NodeChange/RelChange 属性级 diff |
| TD-05 | golangci-lint 未安装 | `brew install golangci-lint` 并集成到 CI |

### 1.3 版本范围

| 纳入 V1 | 延后到 V2 |
|---------|---------|
| Netbox/CMDB/Controller Connector | Kafka 事件流 |
| `extends` 继承 + 多 Label | PostgreSQL 元数据存储 |
| 属性级 Diff (LocalDiff + Cypher Diff) | Impact/RCA/Simulation Engine |
| MetaCache + AuditLog | Gin HTTP API |
| ConnectorFactory + HTTPClient 公共层 | OpenTelemetry + Prometheus |

### 1.4 新增依赖

| 依赖 | 用途 | 引入任务 |
|------|------|---------|
| `golang.org/x/time/rate` | HTTP 客户端限流 | V1-04 |

其余均使用标准库或现有依赖（`neo4j-go-driver`、`yaml.v3`、`viper`、`mcp-go-sdk`）。

### 3.2 设计方案

#### 3.2.1 Connector 接口增强

新增 `Ping()` 方法用于健康检查和连接验证：

```go
type Connector interface {
    Metadata() ConnectorMetadata
    Collect(ctx context.Context, entityType string) ([]Resource, error)
    Stream(ctx context.Context, entityType string) (<-chan Resource, error)
    Ping(ctx context.Context) error  // 新增

```

`ConnectorMetadata` 扩展：

```go
type ConnectorMetadata struct {
    Name        string
    Type        string         // "mock" / "netbox" / "cmdb" / "controller"
    EntityTypes []string
    BaseURL     string         // REST API 基地址
    Timeout     time.Duration  // 请求超时
    AuthType    string         // "none" / "basic" / "token"
}
```

#### 3.2.2 ConnectorFactory 模式

```go
// ConnectorBuilder 工厂函数类型。
type ConnectorBuilder func(name string, cfg map[string]any, entityTypes []string) (Connector, error)

// ConnectorFactory 按 type 查找 Builder 并创建 Connector。
type ConnectorFactory struct {

func (f *ConnectorFactory) RegisterBuilder(connType string, builder ConnectorBuilder)
func (f *ConnectorFactory) Create(entry ConnectorConfigEntry) (Connector, error)
func (f *ConnectorFactory) CreateFromConfig(configPath string, registry *ConnectorRegistry) error
```

`connectors.yaml` 升级：

```yaml

  - name: netbox-prod
```

#### 3.2.3 HTTP 客户端公共层

```go
// HTTPClient 封装认证、重试、限流、超时、分页。
type HTTPClient struct {
    client  *http.Client
    baseURL string
    auth    AuthConfig
    limiter *rate.Limiter
}

// 认证中间件: Token Auth / Basic Auth
// 重试策略: 指数退避 (500ms/1s/2s/4s)，仅 5xx + 网络错误，最多 3 次
```

## 4. 领域二：属性级 Diff

### 4.1 现状分析

| 方法 | 现状 | 局限 |
|------|------|------|
| Cypher `Diff()` | 4 条 Cypher 差集，基于 URI | 只检测节点/关系增删 |
| YAML `LocalDiff()` | 纯内存 Go map 差集 | 同上 |
| `SnapshotDiff` | `AddedNodes/RemovedNodes/AddedRels/RemovedRels` | 无 `ChangedNodes/ChangedRels` |
| `Node.Props` | 已包含完整属性数据 | 数据就绪，未做对比 |

### 4.2 设计方案

#### 4.2.1 新增类型定义

```go
// NodeChange 节点属性级变更。
type NodeChange struct {
    URI            string
    Label          string
    AddedFields    map[string]any          // b 有 a 无
    RemovedFields  map[string]any          // a 有 b 无
    ModifiedFields map[string]FieldChange  // 两边都有但值不同
}

// FieldChange 单个字段的变更详情。
type FieldChange struct {
    OldValue any  // 旧值 (快照 a)
    NewValue any  // 新值 (快照 b)
}

// RelChange 关系属性级变更。
type RelChange struct {
    Type           string
    From           string
    To             string
    AddedFields    map[string]any
    RemovedFields  map[string]any
    ModifiedFields map[string]FieldChange
}
```

#### 4.2.2 SnapshotDiff 扩展

```go
type SnapshotDiff struct {
    AddedNodes   []assembler.Node
    RemovedNodes []assembler.Node
    AddedRels    []assembler.Relation
    RemovedRels  []assembler.Relation
    ChangedNodes []NodeChange  // 新增
    ChangedRels  []RelChange   // 新增
}
```

#### 4.2.3 核心对比逻辑

新增辅助函数 `compareProps(a, b map[string]any)`:
- **数值归一化**: `int(42)` vs `float64(42.0)` 视为相等（统一转 float64 比较）
- **兜底比较**: `fmt.Sprintf("%v", oldVal) != fmt.Sprintf("%v", newVal)`
- **结果分类**: 返回 added / removed / modified 三个 map

**LocalDiff 增强**: 对 URI 交集中的节点调用 `compareProps(aNode.Props, bNode.Props)`。

**Cypher Diff 增强**: 新增第 5 条 Cypher 查找同 URI 但属性不同的节点:
```cypher
MATCH (a {_db: $a}) MATCH (b {_db: $b})
WHERE a.uri = b.uri AND labels(a) = labels(b)
WITH a, b, [k IN keys(a) WHERE a[k] <> b[k]] AS diffKeys
WHERE size(diffKeys) > 0
RETURN a.uri AS uri, labels(a) AS labels,
       properties(a) AS aProps, properties(b) AS bProps
```

## 5. 领域三：本体继承 (extends)

### 5.1 现状分析

| 组件 | 现状 | V1 需要变更 |
|------|------|-----------|
| `EntityTypeSpec` | 无 `Extends` 字段 | 新增 `Extends string` |
| `Node.Label` | `string` 单标签 | 改为 `Labels []string` (BREAKING) |
| Neo4j Cypher | `CREATE (n:Label ...)` | `CREATE (n:Parent:Child ...)` |
| `EnsureIndexes` | 按单 Label 创建索引 | 每个 Label（含父 Label）都需索引 |
| `MERGE` | 单 Label | 只用最具体 Label (Neo4j MERGE 不支持多 Label) |

### 5.2 设计方案

#### 5.2.1 类型层级设计

```
Resource (基类)
├── Device
├── Interface
├── ISIS
├── Link
└── Network_Slice

Service (基类)
├── Network_Slice (可选双继承)

Event (基类)
├── Alarm
```

#### 5.2.2 Schema 扩展

```yaml
# ontology/device.yaml
metadata:
  name: Device
spec:
  extends: Resource        # 继承 Resource 基类
  identity:
    stableKeys: [serial_number]
  uriTemplate: "device:{serial_number}"
```

新增基类 YAML：
- `ontology/resource.yaml`: Resource 基类（通用属性 `status`、`vendor`）
- `ontology/service.yaml`: Service 基类
- `ontology/event.yaml`: Event 基类

#### 5.2.3 继承合并规则

在 `SchemaRegistry.Load()` 后置步骤 `resolveInheritance()` 中执行:

| 字段 | 合并策略 |
|------|---------|
| `Properties` | 子类型同名属性覆盖父类型，其余合并 |
| `FieldMapping` | 子类型覆盖父类型同名映射 |
| `Normalize` | 父类型规则追加到子类型前面 |
| `RelationFields` | 合并（通常不冲突） |
| `Identity` | **不继承**，子类型独立定义 |
| `URITemplate` | **不继承**，子类型独立定义 |
| `Extends` | 支持多级继承（A→B→C），按拓扑序合并 |

**环检测**: DFS 遍历继承链，发现循环返回 error 并阻止启动。

#### 5.2.4 Neo4j 多 Label

**Node 结构变更 (BREAKING)**:
```go
// MVP
type Node struct {
    Label string
    URI   string
    Props map[string]any
}

// V1
type Node struct {
    Labels []string  // ["Resource", "Device"]，从基类到具体类
    URI    string
    Props  map[string]any
}

// 便捷构造函数保持兼容
func NewNode(label string, uri string, props map[string]any) Node {
    return Node{Labels: []string{label}, URI: uri, Props: props}
}
```

**Cypher 生成改造**:
```go
// CREATE: 使用全部 Label
// CREATE (x:Resource:Device {_db: $_db, uri: $uri, ...})

// MERGE: 只使用最具体 Label (Neo4j MERGE 不支持多 Label)
// MERGE (x:Device {_db: $_db, uri: $uri})
// ON CREATE SET x:Resource  // 创建时补充父 Label
// SET x.prop1 = $prop1, ...
```

**SchemaRegistry 新增方法**:
```go
// GetLabels 返回完整标签链（基类→具体类）。
func (r *SchemaRegistry) GetLabels(entityKind string) []string
```

## 6. 领域四：SnapshotService 缓存与审计

### 6.1 现状分析

| 问题 | 现状 | 影响 |
|------|------|------|
| List 性能 | 每次 List 完整解析所有 YAML 文件 | 100 个快照时性能退化严重 |
| 无审计 | 只有 `slog.Info` 级别日志 | 无法追溯操作历史 |
| 薄封装 | 直接委托 SnapshotManager | 无附加价值 |

### 6.2 设计方案

#### 6.2.1 MetaCache

```go
type SnapshotManager struct {
    // ... 现有字段
    metaCache map[string]SnapshotMeta  // name -> meta 缓存
    cacheMu   sync.RWMutex            // 缓存读写锁
}
```

缓存策略:
- `Create()`: 完成后将 meta 写入 cache
- `Delete()`: 从 cache 删除对应条目
- `List()`: 优先从 cache 读取，cache miss 时从 YAML 加载并回填 cache
- 首次 `List()` 触发全量 cache warm（扫描所有 YAML 文件，只解析 meta 文档头）

新增 `importMetaOnly(filePath string) (SnapshotMeta, error)` — 只解码第一个 YAML 文档（meta），不解码 nodes/rels。

#### 6.2.2 AuditLog

```go
// AuditEntry 审计日志条目。
type AuditEntry struct {
    Timestamp time.Time
    Action    string  // "create", "restore", "delete", "diff"
    Snapshot  string  // 快照名称
    Actor     string  // 操作来源 ("mcp", "webhook", "system")
    Detail    string  // 详情 (如 "nodes=21, rels=30")
    Error     string  // 如果有错误
}

// AuditLog 审计日志（内存 FIFO，maxEntries 淘汰）。
type AuditLog struct {
    entries    []AuditEntry
    mu         sync.RWMutex
    maxEntries int
}
```

方法:
- `Record(entry AuditEntry)` — 追加条目，超出 maxEntries 时 FIFO 淘汰
- `Query(filter AuditFilter) []AuditEntry` — 按 Action/Snapshot/时间范围过滤
- `Recent(n int) []AuditEntry` — 最近 N 条

SnapshotManager 集成: 在 Create/Restore/Delete 操作中调用 `audit.Record()`。

#### 6.2.3 TTL 保留策略（可选）

```yaml
# configs/config.yaml
snapshot:
  retention_days: 30  # 0 = 不自动清理
```

`cleanupExpired()`: 扫描 metaCache，`CreatedAt` 超过 retentionDays 的自动 Delete。

## 7. BREAKING CHANGE 迁移指南

### 7.1 Node.Label string → Node.Labels []string

**影响范围**:
- `internal/assembler/types.go` — 核心结构体
- `internal/assembler/assembler.go` — 节点转换
- `internal/graph/neo4j.go` — Cypher 生成（BulkCreate/Upsert/CloneDB/BuildCypher）
- `internal/graph/interface.go` — EnsureIndexes 参数
- `internal/snapshot/exporter.go` — YAML 导出
- `internal/snapshot/importer.go` — YAML 导入（需向后兼容旧格式 `label`）
- `internal/mcp/tools.go` — Diff 输出

**迁移策略**:
1. 新增 `NewNode(label, uri, props)` 便捷构造函数，内部转为 `Labels: []string{label}`
2. `importer.go` 兼容旧 YAML 的 `label` 字段，转为 `[]string{label}`
3. Neo4j `MERGE` 只用最具体 Label（`Labels[len-1]`），`SET` 时补全其他 Label
4. Phase 1 结束时全量回归 `go test -race ./...`

### 7.2 SnapshotDiff 新增字段

**影响范围**:
- `internal/snapshot/manager.go` — SnapshotDiff 结构体
- `internal/mcp/tools.go` — Diff 输出

**迁移策略**: 新增字段 `ChangedNodes`/`ChangedRels` 是追加式，不修改现有字段，向后兼容。

## 8. 风险评估

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| Node.Labels BREAKING CHANGE 波及面广 | 高 | 高 | V1-01 优先完成，提供 `NewNode()` 便捷构造函数，Phase 1 全量回归 |
| 真实 REST API 字段结构不稳定 | 中 | 中 | `transform.go` 隔离字段映射，httptest mock server 做稳定测试 |
| int/float64 类型不一致导致 Diff 误报 | 高 | 中 | `compareProps` 统一归一化为 float64 再比较，测试专门覆盖 |
| Neo4j MERGE 不支持多 Label | 确定 | 中 | MERGE 只使用最具体 Label，ON CREATE SET 补全父 Label |
| 循环继承 | 低 | 高 | `resolveInheritance()` DFS 环检测，发现循环立即返回 error 阻止启动 |
| MetaCache 并发一致性 | 中 | 中 | 复用 SnapshotManager 已有 `sync.Mutex`，cache 操作原子化 |
| Cypher Diff 属性对比性能 | 低 | 中 | 限制单次 Diff 节点数上限，利用 `(_db, uri)` 复合索引 |
