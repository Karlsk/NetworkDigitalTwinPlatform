# V1.2-01: ControllerClient 统一 API 适配层提取

**工时**: 2 天
**前置**: V1 全部完成
**风险等级**: 中（纯内部重构，对外接口不变）
**Phase**: Phase 1 — ControllerClient 提取

---

## 背景

### 问题

当前 `ControllerConnector`（847 行）混合了三层职责：

1. **Token 管理** — `ensureToken()` / `fetchTokenLocked()` / `encryptPassword()`
2. **原始 API 调用** — `fetchDevicesRaw()` / `fetchLinksRaw()` 等 ~20 个方法
3. **Collect + Transform** — `collectDevices()` → `transformDevice()`

这导致：
- MCP 无法直接调用 Controller API（只能走 Collect → 入图 → 查 Neo4j）
- 新增 API 需要在 847 行文件中继续堆积方法
- Token 管理与具体 API 调用紧耦合，无法独立复用

### 目标

提取独立的 `ControllerClient`，封装 **Token 管理 + 所有原始 API 调用**，让 `ControllerConnector.Collect()` 变为薄封装：`client.FetchXxx()` → `transformXxx()`。

## 实现步骤

### Step 1: 创建 `client.go` — ControllerClient 主体

**文件**: `internal/connector/controller/client.go`（新建）

```go
// Package controller 实现 Controller Connector 及统一 API 适配层。
// ControllerClient 封装骨干网操作系统全部 REST API 调用，
// 提供 Token 自动管理 + 类型安全的 Go 方法。
// Connector（拓扑同步）和 MCP（按需查询）共享此 Client。
package controller

// ControllerClient 骨干网操作系统 REST API 统一客户端。
// Token 自动管理（过期刷新 + 60s 缓冲），所有 API 调用前自动 ensureToken。
type ControllerClient struct {
    http         *connector.HTTPClient
    name         string
    // Token 管理
    tokenURL     string
    username     string
    password     string     // 明文密码
    deviceID     string
    desSecretKey string     // 3DES 密钥（24 字节）
    token        string
    tokenExp     time.Time
    // 设备缓存（ISIS/BGP 依赖 vendor 信息）
    cachedDevices []DeviceInfo
    cacheTime     time.Time
    cacheTTL      time.Duration
    mu            sync.Mutex
}

func NewControllerClient(name string, client *connector.HTTPClient, cfg map[string]any) *ControllerClient

// ensureToken 检查 Token 有效性，过期则自动刷新。
// 预留 60 秒缓冲，避免临界过期。
func (c *ControllerClient) ensureToken(ctx context.Context) error

// fetchTokenLocked 调用 POST /oauth/token 获取新 Token。
// 调用方必须持有 c.mu 锁。
func (c *ControllerClient) fetchTokenLocked(ctx context.Context) error

// Ping 验证 Controller 可达性（通过 ensureToken 验证）。
func (c *ControllerClient) Ping(ctx context.Context) error

// getDevices 获取设备列表（优先从缓存读取，TTL 5 分钟）。
func (c *ControllerClient) getDevices(ctx context.Context) ([]DeviceInfo, error)
```

**迁移内容**（从 `controller.go` 移入 `client.go`）：
- `ensureToken()` 方法（L200-211）
- `fetchTokenLocked()` 方法（L213-259）
- `encryptPassword()` 函数（L167-198）
- `getDevices()` 方法（L266-294）

- 设备缓存字段：`cachedDevices` / `cacheTime` / `cacheTTL`

### Step 5: 重构 `controller.go` — 瘦身为 Collect + Transform 编排层

**文件**: `internal/connector/controller/controller.go`（修改）

重构后的 `ControllerConnector` 仅保留：
1. `Connector` 接口方法（`Metadata` / `Collect` / `Stream` / `Ping`）
2. `collectXxx()` 方法（编排 `client.FetchXxx()` + `transformXxx()`）
3. 依赖注入 `ControllerClient`

```go
// ControllerConnector 从控制器 REST API 采集 8 种实体类型数据。
// V1.2 重构：API 调用委托给 ControllerClient，自身仅编排 Collect + Transform。
type ControllerConnector struct {
    client *ControllerClient
    name   string
    types  []string
}

// NewControllerConnector 创建实例（注入 ControllerClient）。
func NewControllerConnector(name string, client *ControllerClient, entityTypes []string) *ControllerConnector

// Collect 全量拉取指定实体类型的数据。
func (c *ControllerConnector) Collect(ctx context.Context, entityType string) ([]connector.Resource, error) {
    switch entityType {
    case "Device":
        return c.collectDevices(ctx)
    // ... 其他 7 种实体类型不变
    }
}

// collectDevices 编排：client.FetchDevices() → transformDevice()
func (c *ControllerConnector) collectDevices(ctx context.Context) ([]connector.Resource, error) {
    rawList, err := c.client.FetchDevices(ctx)
    if err != nil {
        return nil, err
    }
    var resources []connector.Resource
    for _, raw := range rawList {
        props := transformDevice(raw)
        id, _ := raw["id"].(string)
        resources = append(resources, connector.Resource{
            Kind: "Device", ID: id, Properties: props,
        })
    }
    return resources, nil
}
```

## 设计决策

### Q1: 为什么提取到独立文件而不是留在 controller.go？

ControllerClient 未来会被 MCP 层通过 DeviceService 直接调用，而 ControllerConnector 只服务于拓扑同步。分离后：
- `client.go` 可以被 `DeviceService` 直接持有，不需要依赖 Connector 接口
- `api_topology.go` / `api_config.go` 等方法集按 API 类别组织，方便后续扩展

### Q2: Fetch 方法为什么要导出（大写）？

因为 V1.2-02 的 Capability 接口实现会直接调用这些方法。如果保持小写，Capability 接口的实现代码无法跨包访问。

### Q3: ControllerConnector 还需要持有 HTTPClient 吗？

不需要。重构后 `ControllerConnector` 只持有 `ControllerClient`，由 Client 管理 HTTPClient。
