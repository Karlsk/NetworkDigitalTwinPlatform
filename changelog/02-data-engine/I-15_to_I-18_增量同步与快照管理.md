# I-15: SyncService.IncrementalSync + Channel + Webhook

## 1. 任务概述

实现增量同步的三个核心机制：IncrementalSync 方法（处理 update/delete/delete_relation 三种 Action）、Channel 缓冲（防丢消息）、Webhook Handler（立即返回 202）。

| 属性 | 值 |
|------|-----|
| 所属阶段 | Phase 3: 实现阶段 — 同步服务 + 快照管理 |
| 预估工时 | 2 天 |
| 前置任务 | I-14 |
| 交付物 | `internal/service/sync_service.go` 补充 |

## 2. 详细实现步骤

### IncrementalSync 方法

```go
func (s *SyncService) IncrementalSync(ctx context.Context, event SyncEvent) (*SyncResult, error) {
    start := time.Now()

    switch event.Action {
    case "update":
        // 1. 构造 Resource
        resources := make([]connector.Resource, 0, len(event.Data))
        for _, data := range event.Data {
            resources = append(resources, connector.Resource{
                Kind:       event.EntityType,
                Properties: data,
            })
        }

        // 2. Normalizer
        var normalized []normalizer.NormalizedResource
        for _, r := range resources {
            n, err := s.normalizer.Normalize(r)
            if err != nil {
                slog.Warn("normalize failed in incremental sync", "error", err)
                continue
            }
            normalized = append(normalized, *n)
        }

        // 3. Assembler
        model, warnings, err := s.assembler.Assemble(normalized)
        if err != nil {
            return nil, err
        }

        // 4. Upsert (MERGE + SET +=)
        if err := s.graph.Upsert(ctx, "default", model.Nodes, model.Relations); err != nil {
            return nil, err
        }

        return &SyncResult{
            NodesCreated:       len(model.Nodes),
            RelationsCreated:  len(model.Relations),
            OrphanEdgesSkipped: len(warnings),
            Warnings:           warnings,
            Duration:           time.Since(start),
        }, nil

    case "delete":
        if err := s.graph.DeleteByURIs(ctx, "default", event.URIs); err != nil {
            return nil, err
        }
        return &SyncResult{Duration: time.Since(start)}, nil

    case "delete_relation":
        if err := s.graph.DeleteRelations(ctx, "default", event.Relations); err != nil {
            return nil, err
        }
        return &SyncResult{Duration: time.Since(start)}, nil

    default:
        return nil, fmt.Errorf("unknown action: %s", event.Action)
    }
}
```

### Channel 缓冲 + 消费者协程

```go
// StartConsumer 启动消费者协程 (服务启动时调用)
func (s *SyncService) StartConsumer(ctx context.Context) {
    go func() {
        for event := range s.eventChan {
            s.lock.Lock()
            result, err := s.IncrementalSync(ctx, event)
            s.lock.Unlock()

            if err != nil {
                slog.Error("incremental sync failed", "action", event.Action, "error", err)
            } else {
                slog.Info("incremental sync done",
                    "action", event.Action,
                    "nodes", result.NodesCreated,
                    "duration", result.Duration)
            }
        }
    }()
}

// HandleWebhook Webhook Handler (立即返回 202)
func (s *SyncService) HandleWebhook(event SyncEvent) error {
    select {
    case s.eventChan <- event:
        return nil  // 入队成功，返回 202 Accepted
    default:
        return errors.New("event queue full")  // channel 满，返回 503
    }
}
```

## 3. 设计原理

### Channel 缓冲防丢消息

- Restore/FullSync 持有 GraphLock 期间（可能耗时数秒），Webhook 请求到达
- 如果阻塞等锁 → 外部系统超时 → 重试 → 消息重复/丢失
- 解决方案：Webhook → 立即写入 channel → 返回 202 → 消费者协程异步处理

### 单协程消费

- 保证事件顺序（FIFO）
- 避免并发写入冲突
- 简化并发模型

### Webhook Payload 三种 Action

| Action | 节点处理 | 关系处理 |
|--------|---------|---------|
| `update` | MERGE + SET += | MERGE（新增，不删旧关系） |
| `delete` | DELETE 节点 + DETACH DELETE | 节点删除时自动清理所有关联关系 |
| `delete_relation` | 不涉及 | 仅删除指定关系 |

## 4. 验收标准

- [ ] update 事件正确 MERGE 节点和关系
- [ ] delete 事件正确 DELETE 节点
- [ ] delete_relation 事件正确删除指定关系
- [ ] Channel 缓冲：发送 10 个事件后立即返回，异步处理
- [ ] eventChan 满时返回 error（503）

## 5. 注意事项

- eventChan 使用带缓冲的 channel（`make(chan SyncEvent, 100)`）
- 单协程消费保证事件顺序
- Webhook Handler 立即返回 202，不等待处理完成
- 进程重启时 channel 中的事件会丢失（MVP 技术债，V1 用 Kafka）

# I-16: SnapshotManager.Create + List + Delete

## 1. 任务概述

实现快照的基本操作：创建（导出 default DB 为 YAML 归档文件）、列表（扫描 YAML 文件）、删除（清理 Neo4j 逻辑 DB 但保留 YAML）。

| 属性 | 值 |
|------|-----|
| 所属阶段 | Phase 3: 实现阶段 — 同步服务 + 快照管理 |
| 预估工时 | 1.5 天 |
| 前置任务 | I-12, I-13 |
| 交付物 | `internal/snapshot/manager.go`、`internal/snapshot/exporter.go` |

## 2. 详细实现步骤

```go
package snapshot

type SnapshotManager struct {
    graph     graph.GraphDB
    lock      *GraphLock
    snapDir   string
    maxActive int
}

func (m *SnapshotManager) Create(ctx context.Context, name string) (*SnapshotMeta, error) {
    m.lock.RLock()
    defer m.lock.RUnlock()

    // 1. 查询 default DB 全量节点
    nodes, err := m.graph.Query(ctx, "default",
        "MATCH (n {_db: $_db}) RETURN labels(n)[0] AS label, n.uri AS uri, properties(n) AS props", nil)
    if err != nil { return nil, err }

    // 2. 查询 default DB 全量关系
    rels, err := m.graph.Query(ctx, "default",
        "MATCH (a {_db: $_db})-[r]->(b {_db: $_db}) RETURN type(r) AS type, a.uri AS from, b.uri AS to", nil)
    if err != nil { return nil, err }

    // 3. 序列化为 YAML
    meta := SnapshotMeta{
        Name:      name,
        CreatedAt: time.Now(),
        NodeCount: len(nodes),
        RelCount:  len(rels),
        FilePath:  filepath.Join(m.snapDir, name+".yaml"),
    }

    data, err := exportToYAML(nodes, rels, meta)
    if err != nil { return nil, err }

    // 4. 写入文件
    if err := os.WriteFile(meta.FilePath, data, 0644); err != nil {
        return nil, err
    }

    return &meta, nil
}

func (m *SnapshotManager) List(ctx context.Context) ([]SnapshotMeta, error) {
    // 扫描 snapDir 下所有 .yaml 文件，解析元数据
    entries, err := os.ReadDir(m.snapDir)
    if err != nil { return nil, err }

    var metas []SnapshotMeta
    for _, entry := range entries {
        if filepath.Ext(entry.Name()) != ".yaml" { continue }
        // 解析 YAML 头部元数据
        meta, err := parseSnapshotMeta(filepath.Join(m.snapDir, entry.Name()))
        if err != nil { continue }
        metas = append(metas, *meta)
    }
    return metas, nil
}

func (m *SnapshotManager) Delete(ctx context.Context, name string) error {
    // 清理 Neo4j 逻辑 DB（如果存在）
    if has, _ := m.graph.HasDB(ctx, name); has {
        if err := m.graph.ClearDB(ctx, name); err != nil {
            return err
        }
    }
    // YAML 归档文件保留不删除
    return nil
}
```

## 3. 设计原理

- **YAML 是永久归档**：创建后永不删除，Neo4j 逻辑 DB 只是临时缓存
- **Create 使用读锁**：导出期间允许其他读操作，只阻塞写
- **Delete 只清 Neo4j**：YAML 文件保留，随时可以从归档恢复

## 4. 验收标准

- [ ] Create 生成 YAML 文件，含正确的元数据（时间戳、节点数、关系数）
- [ ] List 返回所有 YAML 归档
- [ ] Delete 清理 Neo4j 但保留 YAML
- [ ] YAML 文件可被人工阅读和编辑

# I-17: SnapshotManager.EnsureLoaded + Restore + Diff

## 1. 任务概述

实现快照的懒加载、恢复和对比功能。EnsureLoaded 确保快照已在 Neo4j 逻辑 DB 中加载，Restore 将快照恢复为 default DB，Diff 对比两个快照的差异。

| 属性 | 值 |
|------|-----|
| 所属阶段 | Phase 3: 实现阶段 — 同步服务 + 快照管理 |
| 预估工时 | 1.5 天 |
| 前置任务 | I-16 |
| 交付物 | `internal/snapshot/manager.go` 补充、`internal/snapshot/importer.go` |

## 2. 详细实现步骤

```go
func (m *SnapshotManager) EnsureLoaded(ctx context.Context, name string) error {
    if has, _ := m.graph.HasDB(ctx, name); has {
        return nil // 已加载
    }

    // 从 YAML 文件读取
    filePath := filepath.Join(m.snapDir, name+".yaml")
    nodes, rels, _, err := importFromYAML(filePath)
    if err != nil { return err }

    // 写入 Neo4j 逻辑 DB
    if err := m.graph.BulkCreate(ctx, name, nodes, rels); err != nil {
        return err
    }

    // 触发清理
    return m.cleanup(ctx)
}

func (m *SnapshotManager) Restore(ctx context.Context, name string) error {
    m.lock.Lock()
    defer m.lock.Unlock()

    if err := m.EnsureLoaded(ctx, name); err != nil {
        return err
    }
    if err := m.graph.ClearDB(ctx, "default"); err != nil {
        return err
    }
    return m.graph.CloneDB(ctx, name, "default")
}

func (m *SnapshotManager) Diff(ctx context.Context, a, b string) (*SnapshotDiff, error) {
    if err := m.EnsureLoaded(ctx, a); err != nil { return nil, err }
    if err := m.EnsureLoaded(ctx, b); err != nil { return nil, err }

    // 差集查询
    diff := &SnapshotDiff{}

    // 新增节点：在 b 不在 a
    addedNodes, _ := m.graph.Query(ctx, b,
        `MATCH (n {_db: $_db}) WHERE NOT EXISTS {
            MATCH (m {_db: $other, uri: n.uri}) WHERE labels(m)[0] = labels(n)[0]
        } RETURN n`, map[string]any{"other": a})

    // 删除节点：在 a 不在 b
    // ... 类似逻辑

    return diff, nil
}
```

## 3. 设计原理

- **EnsureLoaded 懒加载**：第一次查 Neo4j → 没有 → 从 YAML 加载 → 后续直接使用
- **Restore 使用写锁**：ClearDB + CloneDB 期间阻塞增量同步
- **Diff 基于 Neo4j 差集查询**：不需要在 Go 代码中比较，直接 Cypher 查询

## 4. 验收标准

- [ ] EnsureLoaded 懒加载：第一次从 YAML 读取，第二次直接使用
- [ ] Restore 恢复 default DB 到快照状态
- [ ] Diff 返回正确的节点/关系差异
- [ ] Restore 期间增量同步被阻塞

# I-18: SnapshotManager.cleanup 懒加载清理

## 1. 任务概述

实现 Neo4j 逻辑 DB 的自动清理：超过 maxActive 个的最久未使用的逻辑 DB 被自动 ClearDB。

| 属性 | 值 |
|------|-----|
| 所属阶段 | Phase 3: 实现阶段 — 同步服务 + 快照管理 |
| 预估工时 | 0.5 天 |
| 前置任务 | I-17 |
| 交付物 | `internal/snapshot/manager.go` cleanup 方法 |

## 2. 详细实现步骤

```go
func (m *SnapshotManager) cleanup(ctx context.Context) error {
    dbs, err := m.graph.ListDBs(ctx)
    if err != nil { return err }

    // 过滤掉 "default"
    var snapshotDBs []string
    for _, db := range dbs {
        if db != "default" {
            snapshotDBs = append(snapshotDBs, db)
        }
    }

    if len(snapshotDBs) <= m.maxActive {
        return nil
    }

    // 按最近使用时间排序（内存 map 维护）
    sort.Slice(snapshotDBs, func(i, j int) bool {
        return m.lastAccess[snapshotDBs[i]].Before(m.lastAccess[snapshotDBs[j]])
    })

    // 清理最旧的
    toClean := snapshotDBs[:len(snapshotDBs)-m.maxActive]
    for _, db := range toClean {
        if err := m.graph.ClearDB(ctx, db); err != nil {
            return err
        }
        delete(m.lastAccess, db)
    }

    return nil
}
```

## 3. 验收标准

- [ ] 超过 maxActive 的最旧逻辑 DB 被自动清理
- [ ] "default" 永远不会被 cleanup 清理
- [ ] cleanup 在 EnsureLoaded 中调用

## 5. 注意事项

- lastAccess 时间戳用内存 map 维护（MVP 不持久化，进程重启后重置）
- 清理策略是 LRU（最近最少使用）
