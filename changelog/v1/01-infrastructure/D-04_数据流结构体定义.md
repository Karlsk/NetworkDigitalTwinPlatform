# D-04: 数据流结构体定义 (Resource / NormalizedResource / GraphModel)

## 1. 任务概述

定义数据流管线中各层之间的数据结构契约：Resource（原始数据）、NormalizedResource（归一化节点）、GraphModel（纯图 IR）。这些结构体是分层解耦的关键，确保每层只关心自己的输入输出。

## 2. 详细实现步骤

### Step 1: Connector 层数据结构

**文件**: `internal/connector/types.go`

```go
package connector

// Resource 原始数据记录 (Connector 输出)
// Kind + ID + Properties，未经任何处理
type Resource struct {
    Kind       string         // 实体类型，如 "Device"
    ID         string         // 原始 ID（数据源内部 ID）
    Properties map[string]any // 原始属性键值对
}

// ConnectorMetadata 连接器元信息
type ConnectorMetadata struct {
    Name        string   // 连接器名称，如 "mock-netbox"
    Type        string   // 连接器类型，如 "netbox", "controller", "cmdb", "mock"
    EntityTypes []string // 支持的实体类型列表
}
```

### Step 2: GraphModel IR 层数据结构

**文件**: `internal/assembler/types.go`

```go
package assembler

// GraphModel 纯图元素的中间表示，与数据源和 Schema 完全解耦
type GraphModel struct {
    Nodes     []Node
    Relations []Relation
}

// Node 图节点
type Node struct {
    Label  string         // 图节点标签，如 "Device"
    URI    string         // 唯一标识 (由 uriTemplate 生成)
    Props  map[string]any // 属性键值对（已类型化，不含原始关系字段）
}

// Relation 图边
type Relation struct {
    Type   string         // 关系类型，如 "HAS_INTERFACE"
    From   string         // 源节点 URI
    To     string         // 目标节点 URI
    Props  map[string]any // 关系属性（可选）
}

// ValidationWarning 图模型校验警告
type ValidationWarning struct {
    Type   string // "orphan_edge"
    Detail string // "HAS_INTERFACE: device:SN12345 → iface:SN12345_GE1/0/2"
}
```

### Step 3: Normalizer 层数据结构

**文件**: `internal/normalizer/normalizer.go`（仅结构体部分）

```go
package normalizer

// NormalizedResource 归一化后的节点记录 (不含关系)
// 关系字段保留在 Properties 中，由 GraphAssembler 读取 relationFields 映射后推导为图边
type NormalizedResource struct {
    Kind       string         // 实体类型，如 "Device"
    URI        string         // 标准本体 URI (由 uriTemplate 生成)
    Properties map[string]any // 标准化后的属性（类型已校验，含原始关系字段）
}
```

### Step 4: Service 层数据结构

**文件**: `internal/service/sync_service.go`（仅结构体部分）

```go
package service

import (
    "time"

    "gitlab.com/pml/network-digital-twin/internal/assembler"
)

// SyncResult 同步结果统计
type SyncResult struct {
    NodesCreated       int
    RelationsCreated  int
    OrphanEdgesSkipped int                            // 孤儿边计数 (可观测)
    Warnings           []assembler.ValidationWarning  // 校验警告
    Duration           time.Duration
}

// SyncEvent 同步事件 (Webhook 触发)
type SyncEvent struct {
    Action     string           // "update", "delete", "delete_relation"
    EntityType string           // 实体类型
    Connector  string           // 连接器名称
    Data       []map[string]any // update 时的数据
    URIs       []string         // delete 时的 URI 列表
    Relations  []assembler.Relation // delete_relation 时的关系列表
}
```

## 3. 设计原理

### 数据流分层 IR 设计

```
Connector → Resource                    (原始数据：Kind + ID + Properties)
         → Normalizer                   (读 Schema → 字段标准化 + URI 生成)
         → NormalizedResource           (归一化节点：Kind + URI + Properties，不含关系)
         → GraphAssembler               (读 relationFields + RelationType → 推导关系)
         → GraphModel (Nodes + Relations) (纯图 IR：Label + Props + Edges)
         → GraphDB                      (封装 Cypher 生成 + 执行)
         → Neo4j
```

每层只关心自己的输入输出，通过显式契约解耦：

- **Resource → NormalizedResource**：Normalizer 读取 Schema 做字段标准化 + URI 生成
- **NormalizedResource → GraphModel**：GraphAssembler 读取 Schema 推导关系
- **GraphModel → Neo4j**：GraphDB 不知道 Schema 的存在，只处理图元素

### NormalizedResource 为什么不直接产出 Relation？

- Normalizer 只处理节点（字段标准化 + URI 生成），不处理关系
- 关系字段保留在 Properties 中传递给 GraphAssembler
- 这样 Normalizer 和 GraphAssembler 职责清晰，各自只读 Schema 的一部分

### GraphModel 为什么与 Schema 解耦？

- GraphDB 只接收 GraphModel，不知道 Schema 的存在
- 修改 Schema（如新增 EntityType）不影响 GraphDB
- 未来替换图数据库只需新建 GraphDB 实现

## 5. 注意事项

- `NormalizedResource.Properties` 中关系字段（如 `interfaces: [...]`）的值是 URI 字符串列表，由 GraphAssembler 读取并推导关系
- `Node.Props` 不应包含关系字段，GraphAssembler 在节点转换时需要过滤
- `Relation.Props` 是可选的，MVP 阶段大部分关系没有属性
- `SyncEvent.Data` 是 `[]map[string]any`，不是 `[]Resource`，因为 Webhook payload 是原始 JSON
- `SnapshotMeta.FilePath` 存储 YAML 归档文件的完整路径，用于后续 List/Delete 操作
- 各结构体之间通过 `internal/assembler` 包的 Node/Relation 类型统一引用
