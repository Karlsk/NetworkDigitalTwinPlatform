# V1.2-04: 监控/日志/切片/SR-TE/确定性网络 API 补全

**工时**: 2 天
**前置**: V1.2-03
**风险等级**: 中（新增大量 API 方法，需逐个对照文档实现）
**Phase**: Phase 4 — API 扩展补全

---

## 背景

### 问题

V1.2-01 提取了 `ControllerClient` 并迁移了已有的 ~10 个 API 方法。V1.2-02/03 定义了能力接口和 MCP 工具，但部分方法仍返回 `ErrNotImplemented`。本任务将 API 文档中剩余 ~30+ 个接口全部补全。

### 目标

按 API 文档（骨干网操作系统API接口规范v1.5）的章节结构，新建 4 个 API 方法集文件，覆盖全部 API 类别：

| 新文件 | 覆盖 API 大类 | 文档章节 | 方法数 |
|--------|-------------|---------|--------|
| `api_monitor.go` | 监控（设备/端口/VPN/Tunnel/告警/日志/拓扑视图） | 6.x | ~12 |
| `api_slice.go` | 切片管理（FlexE/Sub-interface/SRv6 Slice） | 8.x | ~17 |
| `api_srte.go` | SR-TE 路径管理 | 3.x/4.x | ~4 |
| `api_detnet.go` | 确定性网络（DetNet） | 7.x | ~6 |

## 设计决策

### Q1: SR-TE 写操作（路径计算/策略创建）是否实现？

V1.2 仅实现查询类方法（`FetchSRTEPathDetail`），写操作方法（`ComputeSRTEPath` / `CreateSRTEPolicy`）仅预留接口骨架，返回 `ErrNotImplemented`。原因：
- 写操作需要评估安全影响（可能修改网络路径）
- MCP 工具的写操作需要额外的权限控制和确认机制
- V2 引入 HTTP API 后再开放写操作

### Q2: 切片管理 CRUD 是否通过 MCP 暴露？

V1.2 只暴露查询（List）操作通过 MCP。CRUD 中的 Create/Update/Delete 由 `DeviceOperator` 接口预留，但 MCP 工具暂不注册写操作。

### Q3: 监控 API 的时间格式如何处理？

Controller API 文档使用 `"2026-04-21 10:00:00"` 格式，Go 内部使用 `time.Time`。在 `api_monitor.go` 中统一做转换：

```go
func formatMonitorTime(t time.Time) string {
    return t.Format("2006-01-02 15:04:05")
}
```
