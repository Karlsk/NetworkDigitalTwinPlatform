# T-06: SyncService 集成测试

## 1. 任务概述

测试 SyncService 的 FullSync 和 IncrementalSync 完整流程，验证全量同步、增量更新、删除、Channel 缓冲等功能。

| 属性 | 值 |
|------|-----|
| 所属阶段 | Phase 4: 测试阶段 |
| 预估工时 | 1.5 天 |
| 前置任务 | I-14, I-15, T-04, T-05 |
| 交付物 | `internal/service/sync_test.go` |

## 2. 详细测试用例

| 用例ID | 测试内容 | 期望结果 |
|--------|---------|---------|
| TC-S01 | FullSync 全量同步 Mock 数据 | 节点 ≥ 20, 关系 ≥ 30 |
| TC-S02 | FullSync 孤儿边检测 | OrphanEdgesSkipped > 0（注入脏数据） |
| TC-S03 | IncrementalSync update | 属性增量合并（SET +=） |
| TC-S04 | IncrementalSync delete | 节点被 DETACH DELETE |
| TC-S05 | IncrementalSync delete_relation | 关系被删除，节点不受影响 |
| TC-S06 | Channel 缓冲 | 发送 10 个事件全部处理 |
| TC-S07 | URI 不可变性 | 修改 hostname 后 URI 不变 |
| TC-S08 | FullSync 后 Query 验证 | Cypher 查询返回正确数据 |

## 3. 设计原理

- 使用 testcontainers-go 启动 Neo4j + 真实 Schema/Connector/Normalizer/Assembler
- FullSync 测试后验证 Neo4j 中的节点和关系数量
- IncrementalSync 测试需要在 FullSync 之后执行（有基础数据）

## 4. 验收标准

- [ ] 所有测试用例通过
- [ ] `go test -tags=integration -v ./internal/service/...` 无失败

## 5. 注意事项

- 集成测试依赖 Neo4j 容器，确保 Docker 环境可用
- Channel 缓冲测试需要等待消费者协程处理完成（使用 `time.Sleep` 或 `sync.WaitGroup`）
- URI 不可变性测试：FullSync → 修改 hostname → IncrementalSync update → 验证 URI 未变

# T-08: GraphLock 并发测试

## 1. 任务概述

验证 GraphLock 的并发安全性：写锁互斥、读写锁兼容、Restore 期间不丢消息、无数据竞争。

| 属性 | 值 |
|------|-----|
| 所属阶段 | Phase 4: 测试阶段 |
| 预估工时 | 1 天 |
| 前置任务 | I-13, I-15, I-17 |
| 交付物 | `internal/snapshot/graphlock_test.go` |

## 2. 详细测试用例

| 用例ID | 测试内容 | 期望结果 |
|--------|---------|---------|
| TC-L01 | 写锁互斥 | goroutine A 持有 Lock → B 阻塞 |
| TC-L02 | 读写锁兼容 | 多个 RLock 共存 |
| TC-L03 | Restore 期间 Webhook 不丢失 | 发送 10 个事件，全部处理 |
| TC-L04 | FullSync 期间增量等待 | 不报错，阻塞后执行 |
| TC-L05 | 并发读写无竞争 | `go test -race` 无 data race |

## 3. 设计原理

- TC-L03 关键场景：Restore 持有写锁（耗时操作）→ 同时发送 10 个 Webhook → 验证 channel 缓冲全部入队 → Restore 完成后消费者逐个处理
- TC-L05 使用 `-race` 标志检测数据竞争

## 4. 验收标准

- [ ] 所有测试用例通过
- [ ] `go test -race ./internal/snapshot/...` 无 data race

## 5. 注意事项

- 并发测试可能偶尔失败（flaky），使用 retry 或增加等待时间
- Restore 模拟耗时操作用 `time.Sleep`
- TC-L03 需要等待所有事件处理完成后再验证
